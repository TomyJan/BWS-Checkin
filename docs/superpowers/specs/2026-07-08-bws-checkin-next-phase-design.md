# BWS Checkin 下一阶段设计规格

## 背景

当前 MVP 已覆盖登录、二维码、互助组、任务状态、PWA 离线快照和基础创建者管理。下一阶段目标不是扩展成复杂协作平台，而是把活动现场可用性、组管理和生产部署能力补齐，让项目可以在 BW 乐园活动当天稳定使用。

本阶段按 A、B、C 顺序推进：

1. A：现场稳定性。
2. B：组管理与活动运营。
3. C：生产部署与安全。

## 总体原则

- 保持轻量工程化，不引入重型权限系统、复杂后台或外部队列。
- API 继续使用 `/api/v1` 前缀，只使用 `GET` 和 `POST`。
- API 路径继续采用动作式语义，例如 `/group/update`、`/group/archive`。
- HTTP 状态码只表达传输和鉴权状态；业务错误仍通过响应体 `ok` 和 `error.code` 表达。
- 离线同步保持后台自动运行，不增加复杂同步面板、手动同步按钮或冲突弹窗。
- 冲突仍由服务端按 `updated_at` 处理，同一 `group_id + task_id + target_user_id` 状态以更新时间较新的记录为准。
- 前端只在离线时显示合适的「离线模式」徽标，避免现场操作时产生信息噪音。

## A：现场稳定性

### 目标

现场用户进入互助组后，即使网络中断，也能继续切换任务、切换成员、查看二维码并记录打卡状态。系统应尽量在进入页面时完成必要缓存，离线时只给出简洁可理解的状态提示。

### 功能需求

- 进入互助组详情页时，继续缓存完整组快照：组信息、全部任务、成员状态和二维码图片本体。
- 离线状态下，顶部只显示「离线模式」徽标，不显示待同步数量、手动同步按钮、冲突提示或复杂错误详情。
- 已缓存过的互助组，在断网后可以从本地快照打开。
- 离线产生的完成 / 撤销变更继续写入 pending 队列，联网后自动重试同步。
- 二维码图片预缓存失败不阻塞页面使用，但对应成员仍显示缺失状态。
- 底部成员列表应支持按二维码状态辅助现场操作：缺失二维码的成员要有清晰视觉状态。
- 当前成员完成后，可自动切到下一个未完成成员，减少连续打卡时的点击量。
- 点位列表中显示每个点位的完成进度，便于现场快速切换。

### 非目标

- 不做手动同步入口。
- 不做同步队列详情页面。
- 不做冲突提示弹窗。
- 不做多人实时协同推送。

## B：组管理与活动运营

### 目标

创建者可以在活动前和活动中维护互助组信息，控制加入方式，并在活动结束后归档互助组。普通成员仍保持简单体验。

### 功能需求

- 创建者可编辑互助组名称、日期和说明。
- 创建者可锁定或解锁加入。锁定后，邀请码和邀请链接仍可复制，但新成员加入会被拒绝。
- 创建者可归档互助组。归档后：
  - 首页默认隐藏已归档组。
  - 已归档组仍可通过「显示已归档」入口查看。
  - 已归档组不允许新成员加入。
  - 已归档组内不允许继续修改任务完成状态。
- 创建者移除成员前需要二次确认。
- 成员管理区域显示成员二维码状态，便于活动前检查谁还没有上传二维码。
- 普通成员不可编辑组信息、锁定加入、归档组或移除成员。

### 非目标

- 不做加入审批流。
- 不做组所有权转移。
- 不做公开组发现。
- 不做复杂 RBAC。

## C：生产部署与安全

### 目标

让服务端在公网部署时有明确配置、合理默认安全边界和可恢复的数据策略。

### 功能需求

- Session Cookie 使用可配置密钥签名，生产环境禁止使用弱默认密钥。
- Cookie 支持通过配置控制 `Secure`、`SameSite` 和有效期。
- 生产环境关闭 mock 登录时，启动阶段校验 OIDC 必要配置，缺失时返回明确错误。
- 二维码上传校验：
  - 最大 5 MB。
  - 只允许 PNG、JPEG、WebP。
  - 通过图片解码确认文件确实是图片。
  - 替换二维码时删除旧文件。
- 本地文件存储继续作为当前方案，但 README 明确说明生产备份需要同时备份 SQLite 数据库和 uploads 目录。
- 后端记录基础审计日志：
  - 组信息更新、锁定 / 解锁、归档。
  - 成员移除。
  - 任务完成 / 撤销。
  - 二维码上传 / 删除。
- 审计日志先只写入 SQLite，不做前端审计页面。

### 非目标

- 不迁移 PostgreSQL。
- 不接入对象存储。
- 不做后台管理系统。
- 不做数据库在线迁移框架；仍在现有 schema 初始化方式下补兼容迁移。

## API 草案

互助组管理：

- `POST /api/v1/group/update`
- `POST /api/v1/group/join-lock`
- `POST /api/v1/group/join-unlock`
- `POST /api/v1/group/archive`
- `GET /api/v1/group/list?includeArchived=1`

任务状态：

- `POST /api/v1/task/complete`
- `POST /api/v1/task/uncomplete`

任务状态接口保持现有语义，但当互助组已归档时返回业务错误 `group_archived`。

用户二维码：

- `POST /api/v1/me/qr/upload`
- `POST /api/v1/me/qr/delete`

二维码接口保持现有路径，并增强校验和旧文件删除。

## 数据模型调整

`groups` 增加：

- `join_locked`：是否禁止新成员加入。
- `archived_at`：归档时间，空值表示未归档。

新增 `audit_logs`：

- `id`
- `actor_user_id`
- `action`
- `group_id`
- `target_user_id`
- `task_id`
- `metadata`
- `created_at`

已有本地数据库需要兼容迁移。服务启动时应补齐缺失列和缺失表，不要求用户手工删除数据库。

## 前端 UX

- 首页继续以「我的互助组」为第一屏。
- 首页新增「显示已归档」开关，默认关闭。
- 组详情页保留二维码优先、点击显示 / 隐藏 UI 浮层的交互。
- 组管理入口仍放在创建者的更多菜单中。
- 离线时只显示「离线模式」徽标。
- 归档组在详情页显示「已归档」徽标，并禁用完成 / 撤销按钮。
- 缺失二维码成员在成员格中使用清晰但不刺眼的状态标识。

## 测试重点

后端测试：

- 锁定加入后，普通用户无法加入互助组。
- 归档后，无法加入组，也无法修改任务完成状态。
- 普通成员不能编辑、锁定、解锁或归档互助组。
- 更新组信息后，列表和详情返回新值。
- 二维码上传拒绝非图片内容。
- 替换或删除二维码时旧文件被删除。
- 生产环境关闭 mock 登录时，缺少 OIDC 配置会启动失败。
- 审计日志记录关键动作。

前端验证：

- 首页默认隐藏归档组，可切换显示。
- 创建者可进入组管理菜单编辑、锁定和归档。
- 移除成员有二次确认。
- 离线时详情页只显示「离线模式」徽标。
- 归档组禁用任务完成按钮。
- 完成当前成员后自动切换到下一个未完成成员。

## 当前确认结论

- A、B、C 全部进入实现范围，按顺序推进。
- 同步 UI 保持极简，不做手动同步、冲突提示或同步队列详情。
- 冲突处理仍由服务端按更新时间较新的状态覆盖。
- 生产部署能力先补必要安全边界，不扩大到数据库和文件存储架构迁移。
